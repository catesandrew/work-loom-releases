Auth: Role Provider Plugin System

The @workloom/auth package implements a pluggable role resolution system. External identity sources (OIDC, database, LDAP, custom) are registered as providers, and a mapping engine translates external role/group names to internal Workloom roles.

Internal Roles​

Workloom uses four static internal roles with a fixed hierarchy:

RoleLevelTypical Use
owner4Full platform control
admin3Organization administration
editor2Create/edit packages and workspaces
viewer1Read-only access (default)

Resolution Strategy: Merge-All​

All registered providers that can resolve for a given identity run in parallel. Results are merged:

  1. Every provider's externalRoles and groups are unioned into a single list
  2. The mapping engine checks each external role against mapping rules
  3. The highest-privilege matching internal role is returned
  4. If no provider returns a match, defaultRole (typically viewer) is used
  5. A failed provider does not block others (graceful degradation via Promise.allSettled)

RoleProvider Interface​

import type { RoleProvider, RoleProviderIdentity, RoleProviderResult } from '@workloom/auth'

interface RoleProvider {
  readonly id: string
  readonly name: string
  readonly priority: number
  canResolve(identity: RoleProviderIdentity): boolean
  resolve(identity: RoleProviderIdentity): Promise<RoleProviderResult>
  healthCheck?(): Promise<{ healthy: boolean; message?: string }>
}

interface RoleProviderIdentity {
  sub: string
  orgId?: string
  email?: string
  rawCredential?: string
}

interface RoleProviderResult {
  externalRoles: string[]
  groups?: string[]
  metadata?: Record<string, unknown>
  cacheTtl?: number
}

Built-in Providers​

OIDCRoleProvider​

Extracts roles from JWT token claims. Supports string claims, array claims, and Zitadel's nested object format.

import { OIDCRoleProvider } from '@workloom/auth'

const oidc = new OIDCRoleProvider({
  roleClaim: 'workloom_role',      // JWT claim containing role(s)
  groupsClaim: 'groups',           // Optional: claim containing group memberships
})

DatabaseRoleProvider​

Queries user roles from the database via an injectable query interface.

import { DatabaseRoleProvider } from '@workloom/auth'

const db = new DatabaseRoleProvider({
  query: {
    async findRolesByUserAndOrg(userId, orgId) {
      // Your database query here
      return ['developer', 'team-lead']
    }
  },
  cacheTtl: 300,  // seconds (default: 300)
})

LDAPRoleProvider​

Queries LDAP directories for group memberships. Uses dependency injection to avoid bundling an LDAP client library.

import { LDAPRoleProvider } from '@workloom/auth'
import type { LDAPClient } from '@workloom/auth'

const ldapClient: LDAPClient = {
  bind: async (dn, password) => { /* your LDAP bind */ },
  searchGroups: async (userAttr, userVal, searchBase, groupAttr) => {
    // Return group CNs/names
    return ['cn=frontend-devs,ou=groups,dc=corp,dc=com']
  },
  unbind: async () => { /* cleanup */ },
}

const ldap = new LDAPRoleProvider({
  client: ldapClient,
  bindDn: 'cn=service,dc=corp,dc=com',
  bindPassword: process.env.LDAP_BIND_PASSWORD!,
  searchBase: 'ou=groups,dc=corp,dc=com',
  groupAttribute: 'memberOf',   // default
  userAttribute: 'mail',        // default
  cacheTtl: 600,                // seconds (default: 600)
})

Writing a Custom Provider​

import type { RoleProvider, RoleProviderIdentity, RoleProviderResult } from '@workloom/auth'

export class SAMLRoleProvider implements RoleProvider {
  readonly id = 'saml'
  readonly name = 'Corporate SAML'
  readonly priority = 12

  canResolve(identity: RoleProviderIdentity): boolean {
    return !!identity.email
  }

  async resolve(identity: RoleProviderIdentity): Promise<RoleProviderResult> {
    const groups = await this.fetchSAMLGroups(identity.email!)
    return { externalRoles: [], groups, cacheTtl: 900 }
  }

  async healthCheck() {
    // Check SAML metadata endpoint is reachable
    return { healthy: true }
  }

  private async fetchSAMLGroups(email: string): Promise<string[]> {
    // Your SAML assertion parsing logic
    return []
  }
}

Registry Setup​

import { RoleProviderRegistry, OIDCRoleProvider, resolveUserRole, loadMappingConfig } from '@workloom/auth'

// 1. Create and populate registry at app startup
const registry = new RoleProviderRegistry()
registry.register(new OIDCRoleProvider({ roleClaim: 'workloom_role' }))
registry.register(new DatabaseRoleProvider({ query: dbQuery }))
registry.register(new LDAPRoleProvider({ client: ldapClient, ...config }))

// 2. Freeze to prevent runtime mutations
registry.freeze()

// 3. Load mapping configuration
const mapping = loadMappingConfig(process.env.ROLE_MAPPING_CONFIG)

// 4. Resolve a user's role
const role = await resolveUserRole(
  { sub: 'user-123', orgId: 'org-456', email: 'dev@corp.com', rawCredential: jwtToken },
  { registry, mapping }
)

Role Mapping​

The mapping engine translates external role/group names to internal Workloom roles. It supports exact name matching and glob patterns.

Configuration File​

Set ROLE_MAPPING_CONFIG to point to a JSON file:

{
  "mappings": {
    "owner": {
      "externalNames": ["platform-owner", "super-admin"]
    },
    "admin": {
      "externalNames": ["org-admin", "it-admin"],
      "patterns": ["*-lead", "*-manager"]
    },
    "editor": {
      "externalNames": ["developer", "engineer"],
      "patterns": ["*-dev", "*-engineer", "team-*"]
    },
    "viewer": {
      "externalNames": ["contractor", "auditor", "guest"]
    }
  },
  "defaultRole": "viewer",
  "caseSensitive": false
}

Default Mapping​

Without a config file, each internal role maps to itself:

{
  "mappings": {
    "owner": { "externalNames": ["owner"] },
    "admin": { "externalNames": ["admin"] },
    "editor": { "externalNames": ["editor"] },
    "viewer": { "externalNames": ["viewer"] }
  },
  "defaultRole": "viewer"
}

Per-Org Overrides​

Organizations can override the default mapping via the role_mappings database table. At resolution time, org-specific overrides are merged on top of the config file defaults:

-- Example: Acme Corp maps their 'platform-engineer' group to editor
INSERT INTO role_mappings (org_id, internal_role, external_names, patterns)
VALUES ('acme-org-id', 'editor', ARRAY['platform-engineer', 'infra-eng'], ARRAY['acme-*-dev']);

Pattern Syntax​

Patterns use glob-style matching where * matches any sequence of characters:

PatternMatchesDoes Not Match
*-developerfrontend-developer, backend-developerdeveloper, dev
team-*team-alpha, team-backendmy-team, teamwork
*-devfrontend-dev, qa-devdeveloper, devops

Environment Variables​

VariablePurposeDefault
AUTH_ISSUEROIDC provider URL(none — dev mode)
AUTH_AUDIENCEExpected JWT audience(none)
AUTH_ROLE_CLAIMJWT claim containing roleworkloom_role
ROLE_MAPPING_CONFIGPath to mapping JSON file(uses defaults)

Client-Safe Import​

For browser/client code that only needs the role hierarchy:

import { ROLE_HIERARCHY, type WorkloomRole } from '@workloom/auth/roles'

This subpath export contains no Node.js dependencies (fs, etc.) and is safe for client bundles.
